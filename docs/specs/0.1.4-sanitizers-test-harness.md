### 3.2 `cmake/sanitizers.cmake` (new)

```cmake
# cmake/sanitizers.cmake — ASan/UBSan wiring (ADR-001 mitigation #3; docs/04).
# Enabled by the *-debug presets via AERO_ENABLE_SANITIZERS (OFF by default).
# Included from the root CMakeLists BEFORE any add_subdirectory(), so the
# directory-scope flags below reach every project target. vcpkg dependencies
# are NOT affected (they build in their own toolchain invocation); ASan
# tolerates uninstrumented code, so linking them stays safe.

if(NOT AERO_ENABLE_SANITIZERS)
    return()
endif()

if(MSVC)
    # MSVC ships ASan only — /fsanitize=undefined does not exist (spec 0.1.4
    # §5a). ASan is incompatible with /RTC1 (present in CMake's default Debug
    # flags) and with incremental linking, so strip the former and force the
    # latter off. MSVC's linker honors the LAST /INCREMENTAL option, and
    # add_link_options() lands after the built-in Debug link flags.
    add_compile_options(/fsanitize=address)
    string(REPLACE "/RTC1" "" CMAKE_CXX_FLAGS_DEBUG "${CMAKE_CXX_FLAGS_DEBUG}")
    add_link_options(/INCREMENTAL:NO)
elseif(CMAKE_CXX_COMPILER_ID MATCHES "Clang|GNU")
    # AppleClang matches "Clang". -fno-sanitize-recover=all is load-bearing:
    # without it UBSan prints its report and exits 0, and CI stays green
    # (spec 0.1.4 §5e). Frame pointers keep ASan stacks readable.
    add_compile_options(
        -fsanitize=address,undefined
        -fno-omit-frame-pointer
        -fno-sanitize-recover=all
    )
    add_link_options(-fsanitize=address,undefined)
else()
    # Mitigations are enforced, not encouraged (docs/04): an unmapped compiler
    # must fail loudly rather than silently build unsanitized.
    message(FATAL_ERROR "AERO_ENABLE_SANITIZERS=ON but compiler '${CMAKE_CXX_COMPILER_ID}' has no sanitizer mapping (cmake/sanitizers.cmake)")
endif()
```

### 3.12 The red-proof commits (temporary — never reach `main`)

```cpp
// red_proof_test.cpp — DO NOT MERGE (0.1.4 acceptance criterion 6).
// Case order matters: a sanitizer abort kills the process, so UBSan must
// fire before ASan (doctest runs cases in registration order).
#include <doctest/doctest.h>
#include <climits>
#include <cstdlib>

TEST_CASE("red-proof 1: plain failing check") {
    CHECK(false);                       // reds every lane, both configs

    int k = INT_MAX;
    k += 1;                             // aborts on macOS/Linux Debug; silent on MSVC (no UBSan) and in Release
    CHECK(k < INT_MAX);
}

TEST_CASE("red-proof 3: ASan heap-use-after-free") {
    int* p = static_cast<int*>(std::malloc(sizeof(int)));
    *p = 41;
    std::free(p);
    CHECK(*p == 41);                    // ASan aborts here (reached on Windows, where UBSan didn't abort)
}
```

Expected per-lane outcomes (record them on the PR):

| Lane | Debug step | Release step |
|---|---|---|
| macOS / Linux | red — CHECK failure recorded, then **UBSan abort** at case 2 (stacktrace via `UBSAN_OPTIONS`) | red — CHECK failure only (no sanitizers in Release; the UB cases typically pass silently — exactly the gap sanitizers cover) |
| Windows | red — CHECK failure recorded, case 2 passes silently (no UBSan — the documented gap), then **ASan abort** at case 3 | red — CHECK failure only |

An accidentally merged proof commit is impossible to miss (`main` would be permanently red), but criterion 6 requires the reverts land **before** merge.

## 4. Approaches considered & tradeoffs

### 4.1 Sanitizer wiring (D1)

| | A. `cmake/sanitizers.cmake` + option + preset switch *(chosen)* | B. Flags in preset `cacheVariables` (`CMAKE_CXX_FLAGS`) | C. `aero::sanitizers` INTERFACE target |
|---|---|---|---|
| Per-compiler conditionals | ✅ Native CMake `if(MSVC)`… | ❌ Presets can't branch on compiler; needs per-OS duplication in JSON | ✅ Genex-capable |
| `/RTC1` strip on MSVC | ✅ `string(REPLACE)` | ❌ Impossible without wholesale re-declaring `CMAKE_CXX_FLAGS_DEBUG` (fragile duplicate of CMake defaults) | ⚠️ Awkward (target flags can't edit global Debug flags) |
| Coverage guarantee | ✅ Directory-scope: every current & future target | ✅ Global | ❌ Every new target must remember `target_link_libraries(... aero::sanitizers)` — a forgotten link is a **silent** coverage gap |
| Fit with ADR-001 "enforced, not encouraged" | ✅ | ⚠️ | ❌ |

**A chosen.** It also establishes the `cmake/` module directory that later build logic (guards, warnings) will use.

### 4.2 Windows UBSan gap (D2)

| | A. ASan-only on MSVC, documented *(chosen)* | B. clang-cl on Windows for UBSan | C. Skip sanitizers on Windows |
|---|---|---|---|
| docs/04 toolchain policy (MSVC on Windows) | ✅ Honored | ❌ Violated (new toolchain, new vcpkg triplet questions) | ✅ |

## 5. External verification evidence (2026-07-13)

- **(a) MSVC ASan constraints:** MS Learn "AddressSanitizer" (page dated 2026-05-28) — `/fsanitize=address` "isn't compatible with edit-and-continue, incremental linking, and /RTC"; "works with static and dynamic CRTs (`/MD`, `/MDd`, `/MT`, `/MTd`)"; UBSan appears only in the "future sanitizers" feedback list (`/fsanitize=undefined` does not exist in MSVC). Hence D2's flag set and the docs/04 parenthetical.
- **(b) ASan libs/DLL:** MS Learn "asan-building" — `/INFERASANLIBS` on by default (libs auto-linked); since VS 17.7 all CRT flavors use `clang_rt.asan_dynamic-{arch}.dll`, shipped in the MSVC toolset bin directory (on `PATH` in a dev environment). Hence criterion 7 and edge case 4.
- **(c) Runner image:** actions/runner-images `windows-2022` README — `Microsoft.VisualStudio.Component.VC.ASAN` 17.14.36510.44 installed with VS Enterprise 17.14. The ASan toolchain exists on the pinned image; no installer step needed.
- **(d) doctest at the pin:** `git -C vcpkg show HEAD:ports/doctest/vcpkg.json` (submodule == `builtin-baseline` `f87344ca`) → version **2.5.3**, header-only port (`VCPKG_BUILD_TYPE release`), CMake config fixup → `find_package(doctest CONFIG REQUIRED)` / `doctest::doctest`.
- **(e) Recover-mode experiment (this Mac, AppleClang 21):** `-fsanitize=undefined` alone: report printed, **exit 0** → ctest would pass → CI would stay green on UB. With `-fno-sanitize-recover=all`: **exit 134**. With `-fsanitize=address,undefined`: heap-use-after-free caught. Hence D3.

## 6. Edge cases & operational notes

1. **UBSan's default is recover-and-continue (exit 0)** — the single easiest way to build a sanitizer harness that never fails. `-fno-sanitize-recover=all` (D3) is the fix; treat it as load-bearing, never "cleanup".
2. **MSVC `/RTC1` conflict:** CMake's default `CMAKE_CXX_FLAGS_DEBUG` includes `/RTC1`, which errors out combined with `/fsanitize=address`. The module strips it via `string(REPLACE)` (the leftover double-space is harmless). This is MSVC-only, so it **cannot be verified locally on the Mac** — the incompatible combo makes `cl` error loudly, so the Windows Debug CI lane *building at all* is the proof. Edit-and-continue (`/ZI`), the other documented incompatibility, is a non-issue here: with `cmake_minimum_required(3.28)` CMP0141 is NEW and Ninja builds default to ProgramDatabase (`/Zi`).
3. **MSVC incremental linking conflict:** default Debug link flags carry `/INCREMENTAL`; the appended `/INCREMENTAL:NO` wins because MSVC's linker honors the last occurrence and `add_link_options` lands after the built-in flags.
4. **ASan runtime DLL at test time (Windows):** `/MDd` builds load `clang_rt.asan_dynamic-x86_64.dll` at startup. In CI, `msvc-dev-cmd` exports the dev-env `PATH` through `GITHUB_PATH`, which applies to all later steps including ctest → resolves. Locally on Windows, tests must run from a VS developer environment (same rule the README already states for configuring). Fallback if CI disproves this: one extra Windows step appending `dirname $(which cl)` to `GITHUB_PATH`.
5. **MSYS `link` shadow (D7):** first linked target in the repo. CMake usually records the MSVC linker by full path at configure time, so the shadow may never bite — but the `rm -f /usr/bin/link` line is the action README's documented fix, costs nothing, and removes the question. This PR's Windows lane is the empirical proof either way.
6. **`noTestsAction=error`:** if the suite ever registers zero tests (broken discovery, renamed target, empty build), ctest errors instead of reporting success-on-nothing. Harness rot cannot silently green.
7. **vcpkg cache on first CI run:** the manifest edit busts the exact cache key; `restore-keys` prefix-matches the previous cache, so SDL3 archives restore and only doctest (seconds) builds. No cold rebuild expected.
8. **Uninstrumented dependencies:** SDL3 (and all future vcpkg deps) build unsanitized. ASan explicitly tolerates uninstrumented code, and in 0.1.4 nothing links SDL3 anyway. Known future wrinkle on MSVC: STL container annotations can false-positive when instrumented and uninstrumented code exchange `std::vector`/`std::string` across a boundary — escape hatch is `_DISABLE_VECTOR_ANNOTATION`/`_DISABLE_STRING_ANNOTATION`, owned by whichever future task first links a vcpkg static lib into a sanitized target. Not reachable in 0.1.4 (doctest is header-only and compiles inside the sanitized TUs).
9. **LeakSanitizer differs per platform:** bundled-and-default with ASan on Linux/x86-64, unavailable in AppleClang's ASan, absent in MSVC. Leak coverage is therefore an incidental Linux-lane bonus, not a promise; if it ever false-positives on driver/system libs, suppression files are the tool (future task).
10. **doctest uses exceptions internally** (`REQUIRE` aborts a test case via throw). No conflict: docs/04 bans exceptions **across public API boundaries**; tests are not public API, and the future engine code under test stays exception-free.
11. **Sanitized Debug is slower** (~2× compile, 2–3× run). Irrelevant at one trivial test; the cost grows with the engine and is the accepted price of ADR-001 mitigation #3. Tracy (0.1.5) profiles Release-ish builds, unaffected.
12. **Existing local build dirs:** the preset passes `AERO_ENABLE_SANITIZERS=ON` as a `-D` on every configure, so a pre-0.1.4 `build/macos-debug` picks it up on the next `cmake --preset macos-debug` (full rebuild of the one target — fine). No `rm -rf` needed.
13. **`aero_tests` links no engine code yet** — it can't; none exists. From 0.2.1 on, test TUs link the `core` target and the harness needs no structural change (add sources / link libraries in `tests/CMakeLists.txt`).
