## 3. Design — the artifact set (exact content)

### 3.1 `vcpkg.json` (new, repo root)

```json
{
    "$schema": "https://raw.githubusercontent.com/microsoft/vcpkg-tool/main/docs/vcpkg.schema.json",
    "name": "aero-engine",
    "version": "0.1.0",
    "description": "Open-source, cross-platform 3D game engine with an editor and TypeScript or C++ scripting",
    "homepage": "https://github.com/MichaelBallen97/aero-engine",
    "license": "MIT",
    "builtin-baseline": "f87344cac03158cbf1467264565f1fd36b382a24",
    "dependencies": [
        {
            "name": "sdl3",
            "features": [
                { "name": "vulkan", "platform": "linux" }
            ]
        }
    ]
}
```

Notes:

- **`$schema`** gives VSCode JSON validation (same pattern 0.1.1 used for `CMakePresets.json`); vcpkg ignores `$`-prefixed fields by design.
- **`version` mirrors the CMake `project()` version** (0.1.0, provisional). Two places to bump at release time; both are top-of-file and grep-able. Accepted duplication — vcpkg cannot read CMake's version and manifest `version` is informational for a top-level manifest.
- **Platform-qualified dependency features are valid syntax** — verified against the official `vcpkg-schema-definitions.schema.json` (`dependency-feature-object` = `{name, platform}`).
- **Deliberately absent:** `vcpkg-configuration.json` (builtin registry only — nothing to configure), version `overrides` (baseline pins everything), manifest `features` (nothing to gate yet; see §8 deferred items).

### 3.2 `/vcpkg` submodule + `.gitmodules` (new)

- Submodule `vcpkg` → `https://github.com/microsoft/vcpkg.git` (HTTPS, not SSH — anonymous clones and CI must work), checked out at **`f87344cac03158cbf1467264565f1fd36b382a24`**.
- `.gitmodules` sets `shallow = true` so fresh clones fetch only the pinned commit, not vcpkg's full history.

```ini
[submodule "vcpkg"]
	path = vcpkg
	url = https://github.com/microsoft/vcpkg.git
	shallow = true
```

**The invariant this creates: submodule SHA == `builtin-baseline`, always, bumped together in one commit** (procedure in §6.7). The submodule additionally pins the **vcpkg tool itself**: `scripts/vcpkg-tool-metadata.txt` at this commit pins tool release `2026-05-27` with per-platform SHA-512s, which the bootstrap script enforces. Registry, tool, and manifest baseline are therefore all fixed by one SHA — the strongest form of docs/08 R7's "pinned vcpkg baseline" mitigation.

## 4. Approaches considered & tradeoffs

### 4.1 How to acquire vcpkg (D3)

| | A. Git submodule *(chosen)* | B. `VCPKG_ROOT` env var | C. Bootstrap script |
|---|---|---|---|
| Reproducibility | ✅ Registry + tool + baseline pinned by one SHA | ⚠️ Registry pinned by baseline, but per-machine clones drift; a clone **older** than the baseline fails resolution outright | ✅ Pinned |
| Fresh clone → build | ✅ 2 commands (toolchain auto-bootstraps — verified in `vcpkg.cmake` at the pinned SHA: "Bootstrapping vcpkg before install") | ❌ Manual vcpkg install + env var on every machine (none exists today) | ⚠️ 3 commands |
| CI (0.1.3) | ✅ `actions/checkout` with `submodules: true` | ⚠️ Runner's preinstalled vcpkg may predate the baseline → must clone anyway | ⚠️ Script runs in CI too |
| Maintenance | ✅ Standard git mechanism | ✅ None | ❌ Custom `.sh` + `.bat`/`.ps1` to write, test, maintain on 3 OSes |
| Cost | Submodule UX (forgetting `--recurse` → clear "toolchain not found" error); bigger checkout (mitigated: `shallow = true`) | Env-dependent builds | Reinvents submodules, non-standard |

**A chosen.** It is the vcpkg-docs-recommended pattern for manifest mode and the only option where "pin all third-party dependencies reproducibly from day one" includes the package manager itself.

### 4.2 SDL3 `vulkan` feature on Linux (D1) — chain of evidence

1. The `sdl3` port maps feature `vulkan` → CMake option `SDL_VULKAN`, and `vulkan` is **not** in any platform's default features (port manifest + portfile, verified at the pinned SHA).
2. SDL's own CMake compiles the SDL_GPU Vulkan backend **only** when Vulkan video support is enabled (`if(SDL_VIDEO_VULKAN)` gates `src/gpu/vulkan/*` — verified in `libsdl-org/SDL` CMakeLists).
3. SDL_GPU's other backends are D3D12 (needs `dxgi1_6.h` — Windows) and Metal (Apple). **On Linux there is no other backend.**

Therefore plain `"sdl3"` ⇒ SDL_GPU has zero backends on Linux ⇒ task 0.4.x (RHI) would fail in Linux CI with "no GPU device", far from the actual cause. The platform-qualified feature costs nothing (no extra dependencies — SDL loads Vulkan dynamically at runtime) and is correct-by-construction. Alternatives rejected: *plain sdl3* (strict scope, known deferred breakage — pointless debt), *vulkan everywhere* (Windows prefers D3D12, macOS uses Metal — broader than needed). Caveat recorded: the SDL CMake check was read on SDL's `main` branch, not the 3.4.12 tag; the gating structure has been stable across 3.x, and 0.4.x proves it live the moment a GPU device is created in CI.

### 5.2 Re-check command (used at implementation, and after any future baseline bump)

```bash
SHA=f87344cac03158cbf1467264565f1fd36b382a24   # keep equal to builtin-baseline
for p in sdl3 entt joltphysics quickjs-ng spdlog glm enkits doctest fastgltf ktx sdl3-shadercross; do
  gh api "repos/microsoft/vcpkg/contents/ports/$p/vcpkg.json?ref=$SHA" --jq '.content' | base64 -d \
    | python3 -c "import json,sys; d=json.load(sys.stdin); print(d['name'], d.get('version') or d.get('version-semver') or d.get('version-string'))"
done
```

### 5.3 Fallback mechanism for future gaps (documented, deliberately unused)

If a future dependency has no port at the pinned baseline, or lags a needed version, in order of preference:

1. **`overrides` in `vcpkg.json`** — pin a different version that exists elsewhere in the registry's versions database. No new infrastructure.
2. **Overlay port** — `vcpkg-overlays/ports/<name>/` in this repo, wired via `"overlay-ports"` in a then-created `vcpkg-configuration.json`. Full control of the portfile; stays inside the vcpkg dependency flow.
3. **CMake `FetchContent`** pinned to an upstream tag — last resort; bypasses vcpkg, so the license table (docs/01) and dependency-placement invariant (CLAUDE.md) must be re-checked by hand for that library.
4. **Vendoring** — only for single-file libraries (the stb pattern), committed with license header intact.

None of these is created now — no gap exists.

## 6. Edge cases & operational notes

1. **First configure is slow and needs network** — bootstrap downloads the pinned tool (~a few MB), then vcpkg compiles SDL3 from source (minutes). Subsequent configures (including the Release preset right after Debug) reuse vcpkg's default local binary cache. CI caching is 0.1.3's job.
2. **Linux system prerequisites (handoff to 0.1.3):** the sdl3 port's Linux default features (`x11`, `wayland`, `ibus`) require system headers; the portfile names the exact packages: `libx11-dev libxft-dev libxext-dev libwayland-dev libxkbcommon-dev libegl1-mesa-dev libibus-1.0-dev`. The Ubuntu CI lane installs these before configure. (Also in the README section, §3.6.)
3. **Windows prerequisites (handoff to 0.1.3):** vcpkg requires Visual Studio's **English language pack** (documented vcpkg prerequisite; GitHub runner images have it); Ninja needs the VS dev-prompt environment (known since 0.1.1).
4. **Paths with spaces break vcpkg** (microsoft/vcpkg #39394) — this repo was renamed in 0.1.1; the constraint now applies to *contributors'* clone locations. Not restated in README (niche), lives here.
5. **`joltphysics` 5.5.0 vs upstream 5.6.0:** irrelevant until the task that wraps Jolt into the `physics` subsystem — and **that task does not exist yet**: searching all of `docs/tasks/phase-{0..8}.md` (2026-07-13), no epic or task owns the 3D `physics` subsystem, even though phase-7 (7.1) says "wrap Box2D v3 *the way Jolt is wrapped*". Flagged to Michael as a plan gap (outside 0.1.2's scope — same species as 0.1.1's Q4/clang-format gap that became task 0.1.6). Whenever that task exists, its options are: accept 5.5.0, bump the whole baseline, or `overrides`/overlay per §5.3.
6. **Shallow-submodule fallback:** GitHub serves exact-SHA shallow fetches, but if a git version/host misbehaves: `git -C vcpkg fetch --unshallow`, or drop `shallow = true` (worst case: full history, correctness unaffected).
7. **Baseline bump procedure (future maintenance, one commit):**
   ```bash
   git -C vcpkg fetch origin master && git -C vcpkg checkout <new-sha>
   # edit vcpkg.json: "builtin-baseline": "<new-sha>"
   # re-run §5.2 at <new-sha>; reconfigure; then:
   git add vcpkg vcpkg.json && git commit -m "build: bump vcpkg baseline to <new-sha>"
   ```
   Optional idea for 0.1.3 (not created here): a one-line CI check that `git -C vcpkg rev-parse HEAD` equals the manifest's `builtin-baseline`.
8. **Version duplication:** `vcpkg.json` `"version"` mirrors `project(VERSION)`; bump both at release tags (both provisional 0.1.0 today).
