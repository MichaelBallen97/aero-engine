### 3.2 Root `CMakeLists.txt` (exact content)

```cmake
cmake_minimum_required(VERSION 3.28)

project(AeroEngine
    VERSION 0.1.0                        # provisional pre-release version; v1.0.0 tags at Phase 8
    DESCRIPTION "Open-source, cross-platform 3D game engine with an editor and TypeScript or C++ scripting"
    HOMEPAGE_URL "https://github.com/<owner>/aero-engine"   # <owner> = the GitHub account used in step 10; fill during execution
    LANGUAGES CXX
)

# docs/04: out-of-source builds only.
if(CMAKE_SOURCE_DIR STREQUAL CMAKE_BINARY_DIR)
    message(FATAL_ERROR "In-source builds are not allowed. Configure with a preset: cmake --preset <name>")
endif()

# docs/04: C++20 baseline; C++23 features only where Clang, MSVC, and GCC all support them
# (opted into per-feature later, never by raising this global default).
set(CMAKE_CXX_STANDARD 20)
set(CMAKE_CXX_STANDARD_REQUIRED ON)
set(CMAKE_CXX_EXTENSIONS OFF)

# Layer order = dependency order (docs/03). The golden rule is physical here:
# nothing under engine/ or runtime/ may ever reference editor/.
add_subdirectory(engine)
add_subdirectory(runtime)
add_subdirectory(editor)
add_subdirectory(tools)
add_subdirectory(tests)
```

Notes:

- **Minimum version 3.28** — justification: the presets file needs schema ≥ 3 (CMake 3.21) for the `toolchainFile` field 0.1.2 will add and for `condition` objects used below; schema 8 (CMake 3.28) additionally allows the `$schema` field, giving JSON validation in VSCode — the project's primary editor per `docs/04`. 3.28 predates the project start by well over two years and is comfortably below what current Homebrew/winget/apt and GitHub Actions runners ship (CMake 4.x is current in 2026 — verified via cmake.org: schema v12 corresponds to CMake 4.4). It also safely clears CMake 4.x's removal of `cmake_minimum_required(<3.5)` compatibility.
- **No `CMAKE_EXPORT_COMPILE_COMMANDS`, no compiler warnings/flags here** — flags policy (warnings-as-errors etc.) is not specified by any doc yet; adding it now would be inventing convention. Presets set `CMAKE_EXPORT_COMPILE_COMMANDS` (harmless, tooling-only — see 3.4).
- **`project()` VERSION 0.1.0 is provisional** and clearly labeled so; no doc defines a pre-1.0 scheme.

Design notes:

- **Generator: Ninja on all three platforms.** Backed by the doc's one-preset-per-platform/config wording (single-config model); Ninja is the one first-class single-config generator that exists on all three OSes, and it is what 0.1.2's vcpkg flow and 0.1.3's CI runners use most naturally. **Operational consequence on Windows (verified, `cmake-generators(7)`):** command-line build-tool generators require the environment to be pre-configured for the compiler — i.e. configure from a *VS Developer Command Prompt* (or `vcvarsall.bat`); in CI (0.1.3) a dev-cmd setup step handles this. The alternative (Visual Studio multi-config generator) is recorded in §4.
- **`binaryDir: build/<presetName>`** keeps every configuration isolated under one ignorable `build/` root (out-of-source rule, `docs/04`).
- **Conditions** hide foreign-OS presets (`cmake --list-presets` on the Mac shows only `macos-*`), preventing accidental cross-OS configures. Duplicated onto build presets so both stages are gated.
- **What is deliberately absent:** `toolchainFile` (0.1.2 adds it to `base` — one line, one place), sanitizer flags (0.1.4 extends the debug presets), test presets (0.1.4), workflow presets (0.1.3 if CI wants them). The `base` preset exists precisely so those tasks touch one block.
- **`$schema`** URL is the official cmake.org one (verified 2026-07-13).

## 4. Approaches Considered

### A. Configure-only skeleton — stub CMakeLists + `.gitkeep`, zero targets *(recommended)*

- ✅ Matches the deliverable verbatim ("configures successfully") and 0.1.1's subtask list exactly; nothing to throw away later.
- ✅ Later tasks only fill in pre-wired slots; no placeholder source files to migrate or delete when 0.2.x lands real code.
- ❌ `cmake --build` proves nothing until 0.1.4 — mitigated: `project(… LANGUAGES CXX)` already compiles a test program at *configure* time, so toolchain health is proven by configure alone.

### B. Skeleton + one throwaway "hello" executable target

- ✅ CI (0.1.3) would build a real binary immediately.
- ❌ Invents a source file no task asked for, in a directory whose conventions (namespaces, layout) are set by later owning tasks; it would be deleted at 0.1.4. Scope creep in the first commit of a project whose #1 non-technical risk is scope creep (R5).

### C. Ninja Multi-Config (or Visual Studio on Windows) — fewer presets, `--config` at build time

- ✅ One configure per platform; VS generator would avoid the vcvars requirement on Windows.
- ❌ Contradicts the documented model — `docs/04`: "one preset per platform/config" — and mixes single- and multi-config semantics across platforms just as 0.1.4 needs to attach sanitizer flags to *Debug configure presets*. Rejected on the doc's wording.

**Recommendation: A**, with C's Windows caveat recorded (vcvars/dev-prompt requirement) as an operational note for 0.1.3.
