### 3.2 `vcpkg.json` (new, repo root) — spec §3.1 verbatim

```json
{
    "$schema": "https://raw.githubusercontent.com/microsoft/vcpkg-tool/main/docs/vcpkg.schema.json",
    "name": "aero-engine",
    "version": "0.1.0",
    "description": "Open-source, cross-platform 3D game engine with an editor and TypeScript or C++ scripting",
    "homepage": "https://github.com/MichaelBallen97/aero-engine",
    "license": "MIT",
    "builtin-baseline": "f87344cac03158cbf1467264565f1fd36b382a24",
    "dependencies": [
        {
            "name": "sdl3",
            "features": [
                { "name": "vulkan", "platform": "linux" }
            ]
        }
    ]
}
```

Notes (from the spec):
- **`$schema`** gives VSCode JSON validation (the same pattern 0.1.1 used for `CMakePresets.json`); vcpkg ignores `$`-prefixed fields by design.
- **`version` mirrors the CMake `project()` version** (`0.1.0`, provisional). Accepted duplication — vcpkg cannot read CMake's version, and manifest `version` is informational for a top-level manifest. Both are top-of-file and grep-able; bump both at release tags.
- **Platform-qualified dependency features are valid manifest syntax** (spec-sourced: verified against the official schema — `dependency-feature-object` = `{name, platform}`). Rationale for the Linux Vulkan feature (D1) is in spec §4.2: plain `sdl3` leaves SDL_GPU with **zero backends on Linux**, which would surface as a confusing "no GPU device" failure at task 0.4.x far from the real cause.
- **Deliberately absent:** `vcpkg-configuration.json` (builtin registry only), version `overrides` (baseline pins everything), manifest `features` (nothing to gate yet — see §2 OUT-of-scope + spec §8).

### 3.3 `/vcpkg` submodule + `.gitmodules` (new) — spec §3.2

- Submodule `vcpkg` → `https://github.com/microsoft/vcpkg.git` (**HTTPS**, not SSH — anonymous clones and CI must work), checked out at **`f87344cac03158cbf1467264565f1fd36b382a24`**.
- `.gitmodules` sets `shallow = true` so fresh clones fetch only the pinned commit, not vcpkg's full history:

```ini
[submodule "vcpkg"]
	path = vcpkg
	url = https://github.com/microsoft/vcpkg.git
	shallow = true
```

**The invariant this creates: submodule SHA == `builtin-baseline`, always, bumped together in one commit** (bump procedure in spec §6.7). The submodule additionally pins the **vcpkg tool itself** (`scripts/vcpkg-tool-metadata.txt` at this commit → tool release `2026-05-27` with per-platform SHA-512s, enforced by bootstrap). Registry, tool, and manifest baseline are therefore all fixed by one SHA — the strongest form of docs/08 **R7**'s "pinned vcpkg baseline" mitigation.

> Note on HTTPS vs the machine's git protocol: `gh auth status` reports "Git operations protocol: **ssh**" for `MichaelBallen97`, but the submodule URL is deliberately HTTPS (D3 rationale). This is correct and must not be "fixed" to SSH — the submodule must clone anonymously for contributors and CI.

### 3.4 `CMakePresets.json` — one line into the hidden `base` preset — spec §3.3

Insert `"toolchainFile"` between the existing `binaryDir` (current line 10) and `cacheVariables` (current line 11). The resulting `base` preset:

```json
{
    "name": "base",
    "hidden": true,
    "generator": "Ninja",
    "binaryDir": "${sourceDir}/build/${presetName}",
    "toolchainFile": "${sourceDir}/vcpkg/scripts/buildsystems/vcpkg.cmake",
    "cacheVariables": {
        "CMAKE_EXPORT_COMPILE_COMMANDS": "ON"
    }
}
```

All six platform presets inherit it — so Windows/Linux get the toolchain wiring for free (criterion 7). The `toolchainFile` field requires presets schema ≥ 3; the file is already v8. **Nothing else in the file changes** (no touch to the six platform presets, the six build presets, `$schema`, or `cmakeMinimumRequired`).

## 4. Approaches Considered

The one architecturally meaningful fork is **how the project acquires vcpkg** (D3). The other design points (D1 Linux Vulkan feature, D2 baseline SHA, D4 resolution proof) are settled with evidence in spec §4.2–4.4 and are not re-opened here.

### A. Git submodule at `/vcpkg`, pinned to the baseline SHA *(recommended)*

- ✅ **Reproducibility is total** — registry + tool + manifest baseline all pinned by one SHA (spec §3.3).
- ✅ **Fresh clone → configure is 2 commands** — the `vcpkg.cmake` toolchain auto-bootstraps the tool (spec-sourced: verified in `vcpkg.cmake` at the pinned SHA, "Bootstrapping vcpkg before install"). No manual install.
- ✅ **CI-native (0.1.3)** — `actions/checkout` with `submodules: true`.
- ✅ **Standard git mechanism** — nothing custom to maintain.
- ❌ Submodule UX (forgetting `--recurse-submodules` → a clear "toolchain not found" error) and a bigger checkout — mitigated by `shallow = true`.

### B. `VCPKG_ROOT` environment variable pointing at a per-machine vcpkg clone

- ✅ Nothing committed to the repo.
- ❌ **The tool is not pinned and clones drift**; a clone **older** than the baseline fails resolution outright. Requires a manual vcpkg install + env var on every machine — **none exists today** (`VCPKG_ROOT` is unset, verified). CI runners' preinstalled vcpkg may predate the baseline, forcing a clone anyway. Rejected.

### C. Custom bootstrap script (`.sh` + `.bat`/`.ps1`)

- ✅ Pinned if written carefully.
- ❌ Reinvents submodules and adds a non-standard, 3-OS script to write, test, and maintain. Rejected.

**Recommendation: A.** It is the vcpkg-docs-recommended pattern for manifest mode and the only option where "pin all third-party dependencies reproducibly from day one" includes the package manager itself.
